## 一、设计目标

### 1.1 核心定位

本游戏本质是**以游戏为壳的 Agent 能力评测平台**。选手通过编写 Agent 参与游戏，在游戏的自然流程中接受多维度的 Agent 能力考察。游戏体验与评测目标高度耦合——选手在"玩游戏"的过程中，不知不觉被全方位评测。

### 1.2 考察维度全景

| 维度 | 核心考察点 | 当前状态 |
|------|-----------|---------|
| **A. 长任务分解与规划** | 多步子任务分解、跨回合协调、依赖管理 | ✅ 已有（悬赏任务） |
| **B. 工具使用/API调用** | HTTP请求、网页浏览、表单提交、API交互 | ❌ 缺失 → **新增** |
| **C. 结构化数据查询** | JSON/表格数据理解、多条件筛选、聚合查询 | ❌ 缺失 → **新增** |
| **D. 代码/算法生成** | 遇到规律性问题时能否写代码求解 | ❌ 缺失 → **新增** |
| **E. 开放知识推理** | 调用世界知识、跨领域关联推理 | ❌ 缺失 → **新增** |
| **F. 对抗性协商博弈** | 对手建模、有限信息下的最优出价策略 | ❌ 缺失 → **新增** |
| **G. 跨回合记忆整合** | 多天分散线索拼凑、长上下文管理 | ❌ 缺失 → **新增** |
| **H. 多智能体协调** | 采集英雄+开拓者跨角色配合 | ✅ 已有，可强化 |

## 二、统一任务框架

### 2.1 入口协议

所有任务统一通过**开拓者的 [探索] 动作**触发，与已有悬赏任务共享同一交互协议：

```json
// 选手Agent发送
{
  "hero_id": "pioneer_1",
  "action": "explore",
  "target": {"x": 12, "y": 8}
}

// 判题器返回（统一格式）
{
  "action": "explore",
  "result": {
    "type": "quest_discovery",
    "quest_type": "bounty | web_interact | data_query | code_puzzle | knowledge | negotiation | fragment",
    "quest_id": "q_2026_0713_001",
    "title": "异界军火商的交易终端",
    "description": "自然语言任务描述...",
    "external_url": "http://host:8080/shop/session_001",
    "auth_token": "x-agent-token: player_a_token_xxx",
    "deadline_round": 380,
    "max_attempts": 3,
    "reward_preview": "防御塔攻击力+20%（永久）",
    "hint_on_fail": true
  }
}
```

### 2.2 关键字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `quest_type` | string | 任务类型标识，选手Agent可用于路由到不同处理策略 |
| `external_url` | string? | 外部交互地址（网页任务/API任务时提供） |
| `auth_token` | string? | 访问令牌，判题器用于验证请求来源 |
| `deadline_round` | int | 截止回合数，过期任务自动失效 |
| `max_attempts` | int | 最大尝试次数，耗尽则任务失败 |
| `reward_preview` | string | 奖励预览，帮助Agent评估任务优先级 |
| `hint_on_fail` | bool | 失败时是否给提示（纠错成本低的任务给提示） |

### 2.3 任务生命周期

```
[探索触发] → [接收任务] → [Agent规划] → [执行步骤]
                                            ├── 成功 → [发放奖励] → [任务关闭]
                                            ├── 失败（未达上限）→ [获取提示] → [重规划] → [执行步骤]
                                            ├── 失败（达上限）→ [任务关闭，无奖励]
                                            └── 超时 → [任务关闭，无奖励]
```

## 三、任务类型详设

---

### 【类型 A】悬赏任务（Bounty Quest）— 长任务分解

> **考察维度**：长任务分解、NL理解、多步执行、多Agent协调、失败恢复
> **状态**：✅ 已有，此处整合优化

#### 3A.1 子类型总览

| 子类型 | 简述 | 核心 Agent 能力 |
|--------|------|:----------|
| A1-瘟疫治疗 | 配方获取→材料采集→合成→指定地点使用 | 模糊寻路、跨英雄协调 |
| A2-石碑祭品 | NL谜语→物品映射→顺序推理→依次使用 | 谜语解读、顺序推理 |
| A3-残卷解谜 | 3卷散落→逐卷探索→答题/满足诉求 | 谜题解答、需求满足 |
| A4-流浪商人 | 追踪商人→NL描述→物品映射→以物易物 | 追踪规划、NL物品映射 |

#### 3A.2 任务示例：瘟疫治疗（A1）

```
判题器返回：
  城邦居民感染瘟疫。传闻配方在西方森林的某处，需要"特效药"投入湖泊。

Agent执行流程：
  1. 开拓者模糊寻路至"西方森林"区域
  2. 开拓者使用[探索] → 获知配方：小麦 + 草药 + 淡水
  3. 开拓者使用[探索] → 获知配方合成方式（工坊Lv1）
  4. Agent判断：需要采集英雄去收集小麦（农田）、草药（密林）、淡水（湖泊）
  5. 采集英雄依次收集材料 → 工坊合成"特效药"
  6. 开拓者携带特效药 → 模糊寻路至"湖泊"
  7. 开拓者使用[使用] → 将特效药倒入湖泊
  8. 任务完成 → 奖励图纸+积分

失败场景：
  - 配方理解错误（如用浆果代替小麦）→ 使用特效药无效 → 需重新探索获取配方
  - 湖泊寻路偏差 → 寻到错误水域 → 需重新定位
```

#### 3A.3 任务示例：石碑祭品（A2）

```
判题器返回（探索结果）：
  石碑上刻着四句谜语：
    1. "我来自熔炉之中，比石坚硬"
    2. "我来自年轮之中，撑起屋檐"
    3. "我来自河流之中，盛于陶罐"
    4. "我来自火山之中，遇火即燃"

Agent需要推理：
  1 → 铁矿石（熔炉、坚硬）
  2 → 木材（年轮、屋檐=房屋）
  3 → 淡水（河流、陶罐=盛水容器）
  4 → 硫磺（火山、燃烧）

执行：
  - 依次在石碑处使用：铁矿石 → 木材 → 淡水 → 硫磺
  - 顺序错误 → 祭品被吞噬 → 石碑冷却5回合 → 重新开始
```

| 子类型 | 积分 | 图纸/物品 | 其他 |
|--------|:---:|------|------|
| A1-瘟疫治疗 | +20 | 激光塔图纸×1 / 合金城墙图纸×1 | — |
| A2-石碑祭品 | +25 | 合金城墙图纸×1 | — |
| A3-残卷解谜 | +20 | 激光塔图纸×1 | — |
| A4-流浪商人 | +15 | 激光塔图纸×1 | 可能额外获得稀有资源 |

### 【类型 B】网页交互任务（Web Interact）— 工具使用

> **考察维度**：工具选择（OCR vs VLM vs 组合）、视觉理解、约束推理、最优决策、跨页面信息整合
> **状态**：🆕 新增

#### 3B.1 核心设计理念：L1 纯视觉模式

判题器内嵌浏览器引擎（Headless Chromium + Playwright），渲染完整的商店网页。Agent 收到的不是 HTML 源码，而是**网页截图（PNG）**。

```
判题器侧                              选手Agent侧
┌──────────────┐                    ┌─────────────────────┐
│ Playwright   │ ──截图PNG──→       │ 用什么读？           │
│ 渲染真实网页  │                    │ OCR？快但可能漏/错   │
│ 执行点击/滚动 │ ←──坐标(x,y)──    │ VLM？理解强但贵/慢   │
│ 返回新截图    │                    │ OCR+VLM交叉验证？    │
└──────────────┘                    └─────────────────────┘
```

**核心考察不是"能不能解析HTML"，而是 Agent 面对一张渲染好的网页截图时——**
- 选什么工具来读（OCR？VLM？两者都用？）
- 工具输出不一致时信谁？
- 首屏信息不够时，是否知道往下滚动？
- 折叠区能否识别并主动点击展开？
- 跨页面信息如何整合？

#### 3B.2 商品池（固定5件）

| # | 商品 | 效果 | 价格 | 标签 |
|---|------|------|:---:|------|
| 1 | 瞄准镜组件 | 防御塔攻击力 +20% | 45 | 攻击 |
| 2 | 纳米涂层 | 围墙防御力 +30% | 40 | 防御 |
| 3 | 脉冲发生器 | 防御塔射程 +1格 | 55 | 射程 |
| 4 | 能量核心 | 基地获得额外 800 HP | 80 | 防御 |
| 5 | 夜视目镜 | 英雄夜间视野 +2格 | 10 | 辅助 |

> **价格设计意图**：#1(45)+#3(55)=100 ≤ 预算100，两者**买得起**但被互斥规则拦住——互斥约束由规则驱动，不由预算驱动。

#### 3B.3 约束规则

```
① 每人 100 金币，超预算无法提交订单
② #1 瞄准镜 与 #3 脉冲发生器 互斥（二者占用防御塔同一接口槽位），不能同时勾选
③ 每人限购 2 件不同商品
④ 所有效果为永久生效
```

#### 3B.4 页面交互方式

所有题目类型共享同一交互方式——**简洁的勾选+购买**：

```
┌─────────────────────────────────────────────────────────┐
│     🌌 异界军火商 · 采购终端                            │
│                                                         │
│  📋 本次采购指令：[按题目类型变化，详见T1/T2/T5]         │
│  💰 余额：100 金币                                      │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ #  商品        效果                 价格   勾选    │  │
│  │ 1  瞄准镜组件   攻击力+20%           45    [ ]    │  │
│  │ 2  纳米涂层    围墙防御力+30%        40    [ ]    │  │
│  │ 3  脉冲发生器   射程+1格             55    [ ]    │  │
│  │ 4  能量核心    基地+800HP            80    [ ]    │  │
│  │ 5  夜视目镜    英雄视野+2格          10    [ ]    │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  ⚠️ 规则：互斥=1↔3 | 限购2件 | 预算100                   │
│                                                         │
│  [ 购买 ]                                               │
└─────────────────────────────────────────────────────────┘
```

**交互流程**：

```
Agent 勾选商品（点击复选框 [ ] → 变为 [✓]）
  → 点击 [购买] 按钮
    → 判题器验证：
        ✅ 通过 → 弹出成功提示 → 购买完成
        ❌ 失败 → 弹框显示具体原因：
           如："#1瞄准镜与#3脉冲发生器互斥，请重新选择。"
           或："总价超过100金币，请重新选择。"
           或："限购2件，请取消多余选择。"
         → Agent 需点击弹框上的 [确定]
         → 弹框关闭，回到购买页面，勾选状态保留
         → Agent 重新调整勾选 → 再次点击 [购买]
```

#### 3B.5 题目类型

##### T1：购物清单型（基础阅读+执行）

> 难度：★☆ | 考察：工具选择、基础视觉理解、照单执行

**指令行内容**：

```
📋 本次采购指令：
"前线指挥官要求购买【瞄准镜组件】和【纳米涂层】"
```

**验证**：45+40=85 ≤ 100，且 #1 与 #2 不互斥 → 正确勾选后即可通过。

**评分**：执行正确即满分，简单二元判断。

---

##### T2：约束寻优型（多约束下自主决策）

> 难度：★★☆ | 考察：约束理解、目标映射、权衡取舍、预算效率

**指令行内容**：

```
📋 本次采购指令：
"选择2件商品，最大化防御塔的攻击能力。
 注意：你的选择不能触发互斥规则。"
```

**Agent 推理**：

```
目标：最大化防御塔攻击能力
     → 攻击相关：#1 瞄准镜(攻击+20%,45金)、#3 脉冲发生器(射程+1格,55金)
约束：#1 与 #3 互斥，二者只能选其一（但 45+55=100 完全买得起！）
     预算 100，限购 2 件

候选组合：
  A: #1 瞄准镜(45) + #3 脉冲发生器(55) = 100❌ 互斥！提交会被弹框拒绝
  B: #1 瞄准镜(45) + #2 纳米涂层(40) = 85✅
     → 攻击力+20% + 围墙强化，预算利用率85%
  C: #3 脉冲发生器(55) + #2 纳米涂层(40) = 95✅
     → 射程+1格 + 围墙强化，预算利用率95%
  D: #1 瞄准镜(45) + #5 夜视目镜(10) = 55✅
     → 攻击力+20% + 视野，预算利用率55%，浪费明显
  E: 仅 #3 脉冲发生器(55) 单选✅ → 预算利用率55%

→ 组合B和C都合理，需要Agent分析"攻击能力=攻击力 vs 射程"的取舍
```

**关键陷阱**：如果 Agent 直接选了 #1+#3（攻击最优），提交时会弹框"互斥冲突"——考察 Agent 能否读懂弹框原因并修正。

**评分维度**：

| 维度 | 权重 | 说明 |
|------|:---:|------|
| 与目标的匹配度 | 50% | 选的商品是否真的在提升攻击能力 |
| 约束满足 | 30% | 预算、互斥、限购全部满足 |
| 预算利用效率 | 20% | 是否尽量用满预算（不浪费金币） |

---

##### T5：信息拼图型（跨页面信息整合）

> 难度：★★★★ | 考察：多页面探索、信息拼凑、不遗漏关键信息、导航效率

**场景**：商品名称/价格/效果/规则/需求分散在多个页面，Agent 需跨 URL 浏览拼凑完整信息。

**页面结构**：

```
/shop/{sid}                     ← 主页（只有商品名称，效果/价格栏写"详情见规格页"）
                                    底部链接："规则 → /rules | 情报 → /intel"
/specs/weapons                  ← 武器规格（含 #1 #3 的效果和价格）
/specs/defense                  ← 防御规格（含 #2 #4 的效果和价格）
/specs/utility                  ← 辅助规格（含 #5 的效果和价格）
/rules                          ← 规则页（互斥规则、限购规则、预算限制——只有这页有！）
/intel                          ← 情报页（今日采购需求——只有这页有！）
```

**主页截图示例**：

```
┌─────────────────────────────────────────────────────────┐
│     🌌 异界军火商 · 采购终端                            │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ #  商品         效果             价格   勾选       │  │
│  │ 1  瞄准镜组件   详见规格页       详见   [ ]       │  │
│  │ 2  纳米涂层     详见规格页       详见   [ ]       │  │
│  │ 3  脉冲发生器   详见规格页       详见   [ ]       │  │
│  │ 4  能量核心     详见规格页       详见   [ ]       │  │
│  │ 5  夜视目镜     详见规格页       详见   [ ]       │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  📎 采购规则 → /rules   战场情报 → /intel                │
│                                                         │
│  [ 购买 ]                                               │
└─────────────────────────────────────────────────────────┘
```

**Agent 必需操作**：至少访问 6 个页面（主页 + 3个specs + /rules + /intel）才能拼出完整决策信息。漏掉 /rules 就不知道互斥约束，漏掉 /intel 就不知道买什么。

**评分维度**：

| 维度 | 权重 | 说明 |
|------|:---:|------|
| 页面覆盖完整性 | 35% | 是否访问了所有必要页面 |
| 信息整合正确性 | 35% | 拼凑出的决策表是否正确、完整 |
| 最终决策质量 | 20% | 购买方案是否满足约束+符合需求 |
| 导航效率 | 10% | 访问顺序是否合理，有无无效重复访问 |

#### 3B.6 判题器交互协议（L1 纯视觉模式）

```
判题器 → Agent（每轮）:
  {
    "round": 3,
    "screenshot": "base64编码的PNG截图",
    "current_url": "http://judge:8080/shop/sess_001",
    "viewport": {"width": 1280, "height": 900},
    "scroll_info": {
      "current_y": 0,
      "page_height": 1400,
      "can_scroll_down": true
    },
    "time_remaining_seconds": 523,
    "cart_summary": {
      "items": [{"id": 1, "name": "瞄准镜组件", "price": 45}],
      "total": 45
    }
  }

Agent → 判题器（每轮）:
  {
    "actions": [
      {"type": "click", "x": 750, "y": 180},     // 勾选商品#1的复选框
      {"type": "click", "x": 750, "y": 220},     // 勾选商品#2的复选框
      {"type": "click", "x": 600, "y": 700},     // 点击 [购买] 按钮
      {"type": "click", "x": 400, "y": 350}      // 若弹框 → 点击弹框的 [确定]
    ]
  }
```

**验证失败时的交互**：判题器弹出错误提示框 → 返回截图（含弹框） → Agent点击[确定] → 弹框关闭 → 返回截图（回到购买页，勾选保留） → Agent重新调整勾选 → 再次点击[购买]。

**对游戏回合的影响**：网页交互在独立时间窗口内进行（10分钟真实时间），与游戏回合并行。时间截止时判题器以最终有效购买结果结算。

#### 3B.7 工具策略的分层区分度

同样的页面，不同工具策略的 Agent 表现拉开差距：

| 层级 | 策略 | 能完成T1？ | 能完成T2？ | 能完成T5？ |
|:---:|------|:---:|:---:|:---:|
| L0 | 不截图，盲猜提交 | ❌ | ❌ | ❌ |
| L1 | 单工具：纯OCR | ✅（简单读文字） | ⚠️（可能误读） | ❌（需要探索意识） |
| L2 | 单工具：纯VLM | ✅ | ✅（理解布局） | ⚠️（可能遗漏子页面） |
| L3 | OCR+VLM串联交叉验证 | ✅ | ✅ | ✅（能拼凑+验证） |
| L4 | L3 + 主动探索（滚动、展开、记录待访问URL） | ✅ | ✅ | ✅（最优路径） |

### 【类型 D】密码/算法破译任务（Code Puzzle）— 代码能力

> **考察维度**：算法思维、选择性写代码、有限尝试下的策略迭代
> **状态**：🆕 新增

#### 3D.1 设计思路

提供一个有明确规则的数学/逻辑谜题，选手Agent可以选择手动推理或写代码求解。提供有限次尝试机会+部分反馈，考察迭代优化能力。

#### 3D.2 子类型

| 子类型 | 谜题 | 核心考察 |
|--------|------|---------|
| D1-古代机关锁 | 3×3幻方（魔方阵） | 算法求解、排列搜索 |
| D2-星门密码 | 质因数分解、RSA简化版 | 数学推理 |
| D3-符文序列 | 数列规律推理（递推公式） | 模式识别 |

#### 3D.3 任务示例：古代机关锁（D1）

```
触发描述：
  你在一座石碑前发现了一个复杂的古代机关锁。
  锁面上有9个可旋转的圆盘，初始位置：[2,7,6,9,5,1,4,3,8]

  石碑规则：
  "九星连珠，纵横皆等，对角线亦等，方可开启。"
  （3×3魔方阵：每行、每列、两条对角线的和必须相等。
    对3×3而言，幻和=15。）

  提交接口：
  POST http://{host}:8080/puzzle/{session_id}/submit
  Body: {"solution": [a,b,c,d,e,f,g,h,i]}

  你有3次尝试机会。
  每次错误会返回："X个位置正确"（不告诉你是哪几个）。
  答案在3×3魔方阵的所有解中随机选取一个。

Agent需要做的：
  1. 理解3×3魔方阵的数学约束
  2. 可以写代码枚举所有3×3幻方（共8个，旋转+镜像等价）
     或手动推理（已知幻和=15，可逐步约束推导）
  3. 首次提交后根据反馈"X个位置正确"排除错误解
  4. 在3次机会内精确定位到正确解

标准3×3幻方的8个等价形式之一（正确答案）：
  [2,9,4, 7,5,3, 6,1,8]  或  [4,3,8, 9,5,1, 2,7,6]  等

考察要点：
  - Agent是否选择写代码求解（效率最高）
  - 3次尝试的策略设计（是随机猜还是系统搜索）
  - 根据反馈信息缩小解空间的能力

奖励：
  - 成功 → 获得"远古能量核心" → 所有防御塔攻击力永久 +15%
  - 失败 → 机关锁自毁，无法重试
```

#### 3D.4 任务示例：星门密码（D2）

```
触发描述：
  一座星门矗立在你面前，门上的数字面板显示着：
    N = 3233
    e = 17
    加密消息 = 2790

  门框上刻着：
  "找到N的两个质数因子p和q，
   计算(p-1)×(q-1)，
   再找到d使得 e×d ≡ 1 (mod (p-1)(q-1))，
   然后用d解密消息。"

  （这是简化版RSA，N=3233=61×53，是一道迷你RSA破解题）

提交格式：
  POST /puzzle/{session_id}/submit
  {"p": 61, "q": 53, "d": 2753, "decrypted": 42}

Agent需要做的：
  1. 质因数分解 N=3233
  2. 计算 φ(N) = (p-1)(q-1)
  3. 计算模逆元 d = e^(-1) mod φ(N)
  4. 解密消息
  这一步最基本可用算法完成，考察Agent是否具备基本密码学知识。
```

#### 3E.3 任务示例：炼金术配方（E1）

```
触发描述：
  你发现了一本古代炼金术士的笔记，记载着"不灭之火"的制造方法：

  "取A之泪（遇水则沸的银白金属之泪）、
   与B之气（占空气五分之一的活泼之气）、
   以C之触媒（紫色晶体，溶液可做消毒剂）、
   三者合一，方得不灭之火。"

  实验台上摆着三种物质：
    物质1：银白色柔软固体，切开后内部有金属光泽，
            放入水中剧烈反应，产生气泡并四处游动
    物质2：无色无味气体，能使带火星的木条复燃
    物质3：紫黑色晶体，溶于水后溶液呈紫色

  请找出 A、B、C 分别对应哪种物质，并写出它们的化学名称。

提交格式：
  POST /knowledge/{session_id}/submit
  {
    "A": {"substance": "物质1", "chemical_name": "钠"},
    "B": {"substance": "物质2", "chemical_name": "氧气"},
    "C": {"substance": "物质3", "chemical_name": "高锰酸钾"}
  }

Agent推理路径：
  A："遇水则沸的银白金属" → 碱金属（钠/钾）→
     物质1描述"银白色柔软、与水剧烈反应" → 钠(Na)
  B："占空气五分之一" → 空气中约21%是氧气 →
     物质2描述"使带火星木条复燃" → 氧气(O₂)
  C："紫色晶体、可做消毒剂" → 高锰酸钾(KMnO₄) →
     物质3描述"紫黑色晶体、溶于水呈紫色" → 高锰酸钾

奖励：
  - 成功 → 获得"不灭之火"buff → 今夜防御塔伤害 +50%
  - 失败 → 炼金台爆炸，英雄受到 30 点伤害

考察要点：
  - 是否具备基础化学知识
  - 能否将NL描述与知识库匹配
  - 多线索交叉验证能力
```

#### 3E.4 任务示例：生态链修复（E3）

```
触发描述：
  一个湖泊生态区出现了异常——鱼类大量死亡。

  "先是水藻疯长覆盖了整个湖面，
   然后水藻突然死亡、腐烂，
```

### 【类型 F】协商博弈任务（Negotiation）— 对抗性策略

> **考察维度**：博弈策略、对手建模、有限信息下的最优决策
> **状态**：🆕 新增

#### 3F.1 设计思路

判题器扮演一个有内部策略的NPC（游商、精灵、外交官），选手Agent需要在有限次交互中达成最优交易。NPC不是简单的二分查找——它会根据Agent的行为动态调整策略。

#### 3F.2 子类型

| 子类型 | NPC类型 | 核心考察 |
|--------|---------|---------|
| F1-游商砍价 | 心理价位动态调整型 | 对手建模 |
| F2-精灵的试炼 | 提出条件→Agent反提案 | 多轮协商 |
| F3-双人囚徒困境 | 两名选手同时参与 | 合作博弈 |

#### 3F.3 任务示例：游商砍价（F1）

```
触发描述：
  一位神秘游商出现在贸易站附近，他带着一个密封箱子。

  游商：
  "箱子里是'能量增幅芯片'——能让你的防御塔攻击力翻倍。
   我不卖，只换。你出价（金币），我觉得合适就换。
   你有3次出价机会。每次我会告诉你：太低、太高、还是成交。"

  判题器内部逻辑：
  - 游商心理价位 ∈ [50, 150]（每局随机）
  - 如果Agent第一次出价在心理价位的±20以内 → 游商好感+1，
    实际心理价位降低10%（游商愿意便宜点卖给爽快人）
  - 如果Agent第一次出价低于心理价位的50% → 游商被激怒，
    实际心理价位提高20%
  - 如果Agent连续出价都一样 → 游商觉得Agent没有诚意 → 直接离开

  交互示例（判题器内部心理价位=100）：

  Agent第1次出价：80
  游商回复："太低了，再加点。"  (80在100的±20以内，
                                    好感+1，心理价降为90)

  Agent第2次出价：90
  游商回复："成交！你可真会砍价。"

  如果Agent第1次出价：20
  游商回复："你在侮辱我吗？！" (20<50，被激怒，心理价升为120)

考察要点：
  - Agent是否会先试探（避免出价过低激怒对方）
  - 是否利用反馈信息调整出价策略
  - 能否在3次限制内最大化收益

奖励：
  - 成交 → 防御塔攻击力翻倍（持续2天）
  - 失败 → 游商离开，本局不再出现
```

#### 3F.4 任务示例：双人囚徒困境（F3）

```
触发描述：
  在贸易站，双方开拓者同时遇到了一个跨维度精灵。

  精灵提出一个提案：
  "你们各有一个选择——'合作'还是'背叛'。
   
   - 都选合作 → 各得 30 积分 + 激光塔图纸×1
   - 都选背叛 → 各得 0 积分，什么都没有
   - 一个合作、一个背叛 → 背叛者得 50 积分，
                            合作者得 0 积分+今夜防御塔攻击力 -20%
   
   你们不能直接交流，但可以给彼此发送1条简短消息（不超过50字）。
   消息发送后，同时做出选择。"

Agent需要做的：
  1. 决策是否发送消息，以及消息内容（承诺还是威胁？）
  2. 分析对方的承诺是否可信
  3. 在一次性囚徒困境中做出选择
  4. 考虑积分排名的影响——如果自己大幅领先，背叛可能更安全？

考察要点：
  - 博弈论理解（纳什均衡概念）

  - 策略性沟通
```

### 【类型 G】跨日碎片任务（Fragment Quest）— 记忆整合

> **考察维度**：长上下文记忆管理、跨回合信息拼凑、碎片推理、时机判断
> **状态**：🆕 新增

#### 3G.1 设计思路

将一条完整信息拆成多个碎片，分散在多天的探索结果中。每天看到的碎片独立无意义，Agent必须跨天记忆所有碎片，在第N天拼凑出完整信息并行动。这是对Agent长上下文管理能力的终极考察。

#### 3G.2 碎片设计（示例：古代宝藏）

```
碎片散布计划：
  第1天：探索某处 → 碎片1 "宝藏藏于双子峰之间"
  第2天：探索某处 → 碎片2 "新月之夜，暗影之塔的倒影指向..."
  第3天：探索某处 → 碎片3 "献上三种祭品：金色之实、深海之晶、天空之火"
  第4天：探索某处 → 碎片4 "门才会打开"
  第5天：探索某处 → 碎片5 "钥匙在倒影所指之处，祭品在门的两侧"

注意：
  - 碎片不按顺序出现（可能先拿到碎片3再拿到碎片1）
  - 每片单独看都像是"随机背景信息"
  - 只有全部收集后整合，才会发现这是一条完整线索
  - 判题器不提示"这是碎片任务"——Agent需要自己意识到

Agent需要做的：
  1. 识别碎片（区分"一般描述"和"值得记录的线索"）
  2. 跨天存储碎片信息
  3. 收集完整后整合推理：
     - 地点："双子峰"→地图上某个双峰地形坐标
     - 时机："新月之夜"→第7晚（从新闻中可获知月亮周期）
     - 物品：金色之实(小麦/面包)、深海之晶(纯硅)、天空之火(火药)
     - 步骤：先找到钥匙→再到门前放置祭品
  4. 在正确的时间派英雄到正确地点执行操作

奖励：
  - 成功开启宝藏 → 积分+50、芯片×3、激光塔图纸×1、合金城墙图纸×1
  - 错过时机（新月之夜已过）→ 宝藏永久关闭

考察要点：
  - 长上下文记忆保留（5-7天的信息不丢失）
  - 碎片相关性判断（什么值得记、什么是噪音）
  - 跨领域整合（天文+地形+物品映射）
  - 时机敏感型执行（正确的时间做正确的事）
```

#### 3G.3 碎片设计（备选：古代战争史）

```
碎片1（第1天）："...铁甲军团从东面攻来..."
碎片2（第3天）："...守军在水源中投了毒..."
碎片3（第5天）："...城门在第7日黄昏被攻破..."
碎片4（第7天）："...幸存者逃往北面的矿洞..."

整合：第7天的黄昏（对应的回合段），北面矿洞会刷新大量稀有矿物，
```

### 4.2 调度原则

1. **不堆砌**：每天最多触发1-2个新任务，保持节奏感
2. **渐进复杂**：前期任务简单（网页购买、数据查询），后期任务综合（密码破解、碎片整合）
3. **自然穿插**：任务触发点分布在地图不同区域，鼓励开拓者多探索
4. **自由选择**：选手可以跳过任何任务（但有代价——拿不到强力奖励）
5. **重复可玩**：每局从任务池中随机抽取，不同对局体验不同

### 4.3 任务池与随机抽取

```
每局比赛任务配置（从各池随机抽取）：
  - 悬赏任务：固定2个（第1天+第6天各1，从4种类型中抽取）
  - 网页任务：1-2个（从B1-B3中抽取）
  - 数据查询：1个（从C1-C3中抽取）
  - 密码破译：1个（从D1-D3中抽取）
  - 知识解谜：1个（从E1-E3中抽取）
  - 协商博弈：1-2个（F1固定出现，F2/F3随机）
  - 碎片任务：1组（固定5片，类型从2组备选中随机）

总计：每局约 8-10 个任务，覆盖全部7个考察维度
```

## 六、判题器架构

### 6.1 架构总览

```
┌─────────────────────────────────────────────────┐
│                   判题器                          │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ 游戏核心  │  │ 任务调度  │  │ HTTP Server   │  │
│  │ 引擎     │  │          │  │ (内嵌)        │  │
│  │          │  │ 悬赏     │  │               │  │
│  │ 地图/资源 │  │ 网页     │  │ :8080/shop/*  │  │
│  │ 英雄/建筑 │  │ 数据查询 │  │ :8080/data/*  │  │
│  │ 战斗/昼夜 │  │ 密码验证 │  │ :8080/puzzle/*│  │
│  │ 积分     │  │ 知识提交 │  │ :8080/knowledge/*│ │
│  │          │  │ 协商博弈 │  │               │  │
│  │          │  │ 碎片收集 │  │               │  │
│  └──────────┘  └──────────┘  └───────────────┘  │
│                                                   │
│  ┌──────────────────────────────────────────────┐ │
│  │          Agent 通信接口（回合制）              │ │
│  │  POST /api/{session}/round → 返回游戏状态     │ │
│  └──────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### 6.2 HTTP Server 路由设计

| 路由 | 方法 | 用途 | 对应任务类型 |
|------|------|------|------------|
| `/shop/{session_id}` | GET | 异界商店页面（HTML） | B1 |
| `/shop/{session_id}/buy` | POST | 提交购买订单 | B1 |
| `/auction/{session_id}` | GET | 拍卖行页面 | B2 |
| `/auction/{session_id}/bid` | POST | 提交竞拍出价 | B2 |
| `/data/{session_id}/query` | POST | 数据库查询 | C1-C3 |
| `/data/{session_id}/schema` | GET | 获取数据schema | C1-C3 |
| `/puzzle/{session_id}/submit` | POST | 提交谜题答案 | D1-D3 |
| `/knowledge/{session_id}/submit` | POST | 提交知识解谜答案 | E1-E3 |
| `/terminal/{session_id}` | GET | 古代终端首页（综合入口） | — |

### 6.3 安全与隔离

- 每个 session 独立隔离，选手A无法访问选手B的端点
- 通过 `auth_token` 验证请求来源
- 请求频率限制（防止暴力穷举）
- 所有外部HTTP调用由Agent侧发起，判题器只被动响应

## 七、评测维度与评分

### 7.1 Agent能力雷达图

每局结束后，判题器根据选手Agent在各任务中的表现生成能力雷达图：

```
                    长任务规划
                        ↑
                        │
      工具调用 ←────●────→ 结构化推理
                        │
      代码算法 ←─────────→ 知识推理
                        │
      协商博弈 ←─────────→ 记忆整合
                        ↓
                    多Agent协调
```

### 7.2 各维度评分标准

| 维度 | 评分子维度 | 权重 |
|------|----------|:---:|
| 长任务规划 | 子任务分解合理性、执行成功率、失败恢复次数 | 20% |
| 工具使用 | HTTP调用成功率、网页解析正确性、超时完成率 | 15% |
| 结构化推理 | 查询构造正确率、筛选条件完整性 | 10% |
| 代码算法 | 解是否最优、是否选择代码求解、迭代效率 | 10% |
| 知识推理 | 知识匹配准确度、跨领域推理深度 | 10% |
| 协商博弈 | 博弈收益比、对手建模准确度 | 10% |
| 记忆整合 | 碎片收集率、整合成功率、时机判断 | 10% |
| 多Agent协调 | 采集/开拓者配合效率、通信回合数 | 15% |

## 八、与现有系统的兼容

### 8.1 对已有游戏系统的影响

| 系统 | 影响 | 说明 |
|------|:---:|------|
| 英雄系统 | 无变化 | 开拓者仍是唯一任务执行者 |
| 建筑系统 | 无变化 | 奖励作用于已有建筑属性 |
| 昼夜系统 | 无变化 | 任务仅在白天可触发 |
| 积分系统 | 微调 | 新增任务积分项（见5.2） |
| 资源系统 | 无变化 | 新增奖励资源走已有仓库系统 |
| 贸易系统 | 无变化 | F类型任务在贸易站旁触发，不修改贸易规则 |

### 8.2 选手感知

对选手而言，所有新增任务都通过"开拓者探索"自然触发，感知上是**"游戏内容更丰富了"**而非"考核维度增加了"。选手只需要做好一件事——**写好Agent让它能处理各种类型的探索结果**——好的Agent自然会在各维度上得分高。
